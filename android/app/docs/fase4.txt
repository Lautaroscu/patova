ESPECIFICACIÓN TÉCNICA: FASE 4 - RESILIENCIA, PROTECCIÓN CONTRA ABUSO Y LOGISTICA DELTAObjetivo de la FaseTransformar el backend en un sistema robusto preparado para producción masiva. Implementaremos protección contra ataques de denegación de servicio en el endpoint público de Android, un sistema de actualizaciones incrementales (deltas) para optimizar el consumo de datos en móviles y un worker de limpieza para mantener la base de datos Postgres ágil.📋 Tareas Específicas para el Agente4.1. Endpoint de Actualizaciones Delta (GET /v1/downloads/callkit/delta)Problema: Descargar la base de datos completa comprimida en Gzip es eficiente al principio, pero cuando el sistema escale a millones de filas, consumir todo el archivo repetidamente por red móvil destruirá el plan de datos y la batería de los usuarios de iOS.Solución: El agente debe implementar unendpoint delta que acepte un parámetro de fecha opcional (?since=2026-06-13T00:00:00Z).Lógica: La query asíncrona debe consultar únicamente los números cuya fecha en detected_at sea mayor a since. Debe retornar dos estructuras: una lista de números agregados (added) y una lista de números que dejaron de ser spam (removed) si aplica.4.2. Rate Limiting y Protección de Abuso en el Lote de AndroidProblema: El endpoint público POST /v1/reports/batch es la puerta de entrada de los reportes colectivos desde los dispositivos Android. Si un actor malicioso o un bot infectado inunda este endpoint con números falsos o aleatorios, nuestro algoritmo expandirá cada uno en bloques de 1.000, inyectando miles de millones de filas basura en Postgres.Solución: El agente debe integrar un middleware de rate limiting (como slowapi utilizando Redis o memoria local) para limitar la cantidad de solicitudes por dispositivo/IP en un intervalo de tiempo.Validación: Opcionalmente, preparar el esquema de inyección para validar que los payloads no superen un máximo razonable de números reportados por batch (ej: máximo 50 números por lote).4.3. Worker asíncrono de Evicción y Retención de Datos (Data Pruning)Problema: El ecosistema de spam es dinámico; los spammers abandonan o dan de baja rangos de telecomunicaciones cuando se vuelven ineficaces. Mantener registros predictivos de hace años sin actividad ensucia los índices de Postgres.  Solución: Crear una tarea en background (FastAPI BackgroundTasks o un script CRON independiente) que se ejecute periódicamente para limpiar la base.Regla de negocio:Eliminar registros con is_predicted = True que lleven más de 6 meses creados y que no registren ningún impacto (hit) en nuestra tabla blocked_calls_log. Esto garantiza que los números predictivos que nunca llegaron a realizar spam efectivo sean purgados del sistema de forma segura.📝 Plan de Verificación de la Fase 4El agente debe escribir pruebas de integración en tests/test_production_resilience.py utilizando httpx.AsyncClient para comprobar:Que el endpoint /downloads/callkit/delta filtre correctamente por fecha y devuelva solo los registros nuevos generados en los mocks de prueba.Que los intentos de inundación al endpoint /reports/batch gatillen correctamente un código de estado HTTP 429 Too Many Requests.Que la tarea de limpieza purgue de forma efectiva los números predictivos antiguos que no tienen logs de intercepción asociados.
